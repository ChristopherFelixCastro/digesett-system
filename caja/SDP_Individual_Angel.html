Tu Rol en el Sistema

¿Qué eres exactamente?

Tu módulo es la cara operativa del sistema en el mundo físico. Mientras Christopher maneja el portal web para los ciudadanos, tú manejas las dos situaciones donde el agente o cajero trabaja cara a cara con la realidad: en la calle levantando multas y en la ventanilla cobrándolas.

Lo que hace único y complejo tu módulo es que debe funcionar aunque no haya internet. Un agente de tránsito no puede detenerse en el medio de la Avenida 27 de Febrero porque el servidor no responde. Tu app guarda todo localmente en una base de datos cifrada y lo envía al sistema central cuando recupera señal — completamente automático, sin que el agente haga nada extra.

03

Base de Datos Local — SQLite + SQLCipher

El mecanismo de cifrado paso a paso

Esta es la parte más crítica de tu módulo en términos de seguridad. La clave de cifrado nunca se guarda en disco — se recrea cada vez que el agente inicia sesión.

1
El agente ingresa su password en la pantalla de login
Capturas el password en texto plano temporalmente en memoria. Nunca lo guardes en ningún archivo ni variable global.
2
Obtienes el ID único del dispositivo
Usa la dirección MAC de la tarjeta de red principal o el número de serie del procesador como sal única por dispositivo.
3
Derivas la clave con PBKDF2
Combinas password + ID del dispositivo para generar 32 bytes (256 bits) de clave. La clave vive solo en memoria RAM mientras la app está abierta.
4
Abres la BD SQLite con esa clave
La clave se pasa al string de conexión de SQLCipher. SQLite abre o crea el archivo cifrado. Si la clave es incorrecta, la BD no abre — los datos son ilegibles.
5
Al cerrar la app, la clave desaparece
Como la clave solo existía en memoria RAM y no se guardó en disco, al cerrar la app o apagar el dispositivo, la clave se volatiliza. El archivo SQLite queda ilegible hasta el próximo login.

Código de derivación de clave en C#

using System.Security.Cryptography;
using System.Net.NetworkInformation;

private static string DerivarClave(string password)
{
  // Obtener ID único del dispositivo (dirección MAC)
  var mac = NetworkInterface.GetAllNetworkInterfaces()
    .Where(n => n.OperationalStatus == OperationalStatus.Up)
    .Select(n => n.GetPhysicalAddress().ToString())
    .FirstOrDefault() ?? "DIGESETT_FALLBACK_SALT";

  // Derivar clave con PBKDF2 — 100,000 iteraciones (estándar NIST)
  var claveBytes = Rfc2898DeriveBytes.Pbkdf2(
    password:    System.Text.Encoding.UTF8.GetBytes(password),
    salt:        System.Text.Encoding.UTF8.GetBytes(mac),
    iterations:  100_000,
    hashAlgorithm: HashAlgorithmName.SHA256,
    outputLength:  32  // 256 bits para AES-256
  );

  // Convertir a string hexadecimal para pasarlo a SQLCipher
  return Convert.ToHexString(claveBytes);
}

// Abrir la BD con la clave derivada
private static string ObtenerConnectionString(string clave)
{
  var dbPath = Path.Combine(
    Environment.GetFolderPath(Environment.SpecialFolder.ApplicationData),
    "Digesett", "actas_local.db"
  );
  return $"Data Source={dbPath};Password={clave};";
}

Flujo Offline Completo — Paso a Paso

Los cuatro estados de una acta en tu app

✏️
BORRADOR (editable)
El agente está llenando el formulario. Aún no ha presionado "Firmar y Emitir". El registro existe en SQLite con timestamp_firma = NULL. Se puede editar o eliminar libremente. El agente puede salir de la pantalla y volver — el borrador persiste.
🔒
PENDIENTE (firmado, esperando red)
El agente presionó "Firmar y Emitir". El registro en SQLite recibe el timestamp_firma (hora exacta del dispositivo), el UUID definitivo, y el estado_sync = 'PENDIENTE'. Ya no se puede editar. El Network Listener lo enviará automáticamente cuando detecte internet.
✅
ENVIADO (sincronizado correctamente)
El Network Listener envió el acta a Yeimi y recibió confirmación. El estado_sync cambia a 'ENVIADO'. En el HistorialActasForm aparece con un ícono verde de verificación. El acta ya vive en el Core de Antony y es visible en el portal web de Christopher.
⚠️
ERROR (rechazado por el servidor)
Yeimi rechazó el acta al sincronizar (error de validación en el Core). El estado_sync cambia a 'ERROR' y el error_detalle guarda el mensaje de error. En el HistorialActasForm aparece con ícono de advertencia. El agente no se bloquea — puede seguir levantando otras actas. El supervisor lo revisa en el Backoffice.

Implementación del Network Listener

El Network Listener es un hilo en segundo plano que corre mientras la app está abierta. Se implementa con Task.Run() o un BackgroundWorker de WinForms:

// En el MenuPrincipalForm, al iniciar la app después del login:
private async void IniciarNetworkListener()
{
  await Task.Run(async () => {
    while (appActiva)
    {
      // Verificar conectividad cada 15 segundos
      await Task.Delay(15_000);

      bool hayRed = NetworkInterface.GetIsNetworkAvailable();
      if (hayRed)
      {
        await SincronizarPendientesAsync();
        await ActualizarCatalogoInfraccionesAsync();
      }

      // Actualizar el indicador visual en el Form (hilo UI)
      Invoke(() => ActualizarIndicadorRed(hayRed));
    }
  });
}

private async Task SincronizarPendientesAsync()
{
  // 1. Leer pendientes de SQLite
  var pendientes = ObtenerActasPendientes(); // WHERE estado_sync = 'PENDIENTE'

  // 2. Enviar cada una a Yeimi en orden cronológico
  foreach (var acta in pendientes.OrderBy(a => a.TimestampFirma))
  {
    try {
      var json = JsonSerializer.Serialize(acta.ToApiDto());
      var respuesta = await httpClient.PostAsync(
        "/api/v1/actas",
        new StringContent(json, Encoding.UTF8, "application/json")
      );

      if (respuesta.IsSuccessStatusCode || respuesta.StatusCode == HttpStatusCode.Accepted)
        ActualizarEstadoSync(acta.Id, "ENVIADO");
      else
        ActualizarEstadoSync(acta.Id, "ERROR", await respuesta.Content.ReadAsStringAsync());
    }
    catch (Exception ex) {
      ActualizarEstadoSync(acta.Id, "ERROR", ex.Message);
    }
  }
}

El JSON que envías a Yeimi al sincronizar

Este es el formato exacto del body del POST /api/v1/actas que envías. Coordínalo con Yeimi para que coincida con el openapi.yaml:

{
  "id":                     "f4a3b2c1-uuid-generado-en-dispositivo",
  "conductor_cedula":        "001-1234567-8",        // nullable
  "tipo_infraccion_codigo":  "L63-005",
  "placa":                   "A123456",               // nullable
  "fecha_hecho":             "2025-09-15T14:23:00Z",   // timestamp del dispositivo
  "url_evidencia":           "https://...",           // nullable
  "descripcion_vehiculo":    "Toyota Corolla plateado", // nullable
  "grua_numero":             "GRU-045",               // nullable, si retención

Manejo del HTTP 202 Accepted de Yeimi: Cuando el Core de Antony está caído, Yeimi responde con HTTP 202 en lugar de 200. En tu código, trata el 202 igual que el 200 — marca el acta como ENVIADO. La multa quedó en la cola de Yeimi y llegará al Core eventualmente. No necesitas preocuparte por eso.

Actualización del Catálogo de Infracciones

¿Por qué necesitas el catálogo en SQLite?

Cuando el agente está en la calle sin internet, necesita poder seleccionar el tipo de infracción del formulario. Si el catálogo solo viviera en el servidor, la app no funcionaría offline. Por eso guardas una copia local en Infracciones_Cache.

1
Al iniciar sesión con red disponible
Llamas a GET /api/v1/infracciones vía Yeimi para obtener el catálogo actualizado. Borras la tabla Infracciones_Cache y la vuelves a llenar con los datos frescos. Esto asegura que siempre tengas los montos correctos.
2
Cuando el Network Listener detecta reconexión
Además de sincronizar las actas pendientes, también llamas al endpoint del catálogo para actualizarlo. Esto asegura que si Antony agregó nuevas infracciones mientras el agente estaba offline, las tenga disponibles al reconectarse.
3
Si nunca hay internet al iniciar
Usas los datos que ya existen en Infracciones_Cache de la última sincronización exitosa. Si la tabla está vacía (primer uso sin red), muestras un mensaje informativo y deshabilitas el formulario de nueva acta hasta obtener el catálogo.
✅ Coordinación con Antony: Antony debe tener el endpoint GET /api/v1/infracciones disponible desde la Semana 3. Asegúrate de que el formato de respuesta incluya todos los campos que necesitas para tu tabla Infracciones_Cache: código, descripción, categoría, montos por tipo de vehículo y el flag de retención.
